FILE SYSTEM AND LINK ANALYSIS REPORT


1. Command: echo "Important Data" > original.txt
Explanation: I created a new text file containing a string of data. This serves as the target file for our linking experiment.


2. Command: ln original.txt hardlink.txt && ln -s original.txt symlink.txt
Explanation: I used "ln" to create a hard link, which points directly to the original file's inode on the hard drive. I then used "ln -s" to create a symbolic (soft) link, which acts as a shortcut pointing to the file name.


3. Command: ls -li
Output: total 16
        46410 drwxr-xr-x 4 swapnil swapnil 4096 Sep 14  2025 Desktop
         1600 -rw-r--r-- 2 swapnil swapnil   15 Jun 23 20:32 hardlink.txt

         1601 lrwxrwxrwx 1 swapnil swapnil   12 Jun 23 20:32 symlink.txt -> original.txt
Explanation: I ran "ls -li" to view the inode numbers of all three files. I observed that "original.txt" and "hardlink.txt" share the exact same inode number, while "symlink.txt" has a different, unique inode number.

5. Command: rm original.txt && ls -li
Output: swapnil@Swapnil-Patil:~$ rm original.txt
        swapnil@Swapnil-Patil:~$ ls -li
        total 12
        46410 drwxr-xr-x 4 swapnil swapnil 4096 Sep 14  2025 Desktop
         1600 -rw-r--r-- 1 swapnil swapnil   15 Jun 23 20:32 hardlink.txt
         1597 drwxr-x--- 3 swapnil swapnil 4096 Jun 23 20:21 secure_workspace
         1601 lrwxrwxrwx 1 swapnil swapnil   12 Jun 23 20:32 symlink.txt -> original.txt
Explanation: I deleted the original file using "rm" and re-checked the directory. The original file is gone, but the hard link and symbolic link remain.


6. Command: cat hardlink.txt && cat symlink.txt
Output: swapnil@Swapnil-Patil:~$ cat hardlink.txt
        Important Data
        swapnil@Swapnil-Patil:~$ cat symlink.txt
        cat: symlink.txt: No such file or directory
Explanation: I attempted to read both links. The "cat hardlink.txt" command successfully output "Important Data" because the hard link still points to the physical data on the disk. The "cat symlink.txt" command failed and reported a "No such file" error because the shortcut is broken now that the target filename is gone.


Conclusions:
Hard links act as alternate names for the same underlying data block (inode), so the data is preserved even if the original file name is deleted. Symbolic links are merely pointers to a file path; if the target is deleted, the symbolic link becomes orphaned and useless.
